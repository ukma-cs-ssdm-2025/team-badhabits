## 4. Співпраця та робочий процес
- Стратегія гілкування: 
  - GitHub Flow: основна гілка `main`, feature гілки типу `feature/назва-функції`
  - Захист гілки `main` налаштовано через GitHub Rulesets
  - Всі зміни тільки через Pull Requests з обов'язковим code review

- Практики комітів: 
  - Використання англійської мови
  - Формат: `<type>: <description>` (наприклад: `feat: add user login`, `fix: resolve validation issue`)
  - Описові повідомлення, що пояснюють суть змін

- Правила огляду коду:
  - Мінімум 1 схвалення від члена команди для злиття PR
  - Скасування старих схвалень при додаванні нових комітів
  - Вирішення всіх коментарів перед злиттям обов'язкове

- Робочий процес завдань:
  - Використання GitHub Issues для трекінгу завдань
  - Налаштовано систему міток (labels) для категоризації
  - Зв'язування PR з Issues через ключові слова (Fixes #123)
  - CI pipeline перевіряє всі зміни автоматично
