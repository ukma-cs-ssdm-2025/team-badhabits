## 4.1. Співпраця та робочий процес -  Організація середовища GitHub

- Стратегія гілкування: 
  - GitHub Flow:
    - Основна гілка `main` - завжди стабільна і готова до розгортання.
    - Присутні feature гілки: (`feature/feature-name`) для нових функцій.
    - Присутні bug fix гілки (`fix/bug-name`) для виправлень.
  - Захист гілки `main` налаштовано через GitHub Rulesets.
  - Зміни гілки `main` проводяться тільки через **Pull Requests** з обов'язковим <br> **code review**.

- Практики комітів: 
  - Для найменування та опису комітів використовується лише англійська мова.
  - Формат `<type>: <description>` (наприклад: `feat: add user login`, <br> `fix: resolve validation issue`, `docs: update API documentation`, <br> `refactor: simplify database queries`).
  - Для будь-яких дій (змін) повинні бути додані лаконічні описові повідомлення, <br> що пояснюють їх суть.

- Команди **Git** для розробки:
```bash
  # Створення нової feature гілки
  git checkout main
  git pull origin main
  git checkout -b feature/your-feature-name

  # Робота з змінами
  git add .
  git commit -m "feat: add your feature description"
  git push origin feature/your-feature-name

  # Після злиття PR - очищення
  git checkout main
  git pull origin main
  git branch -d feature/your-feature-name
```

- Правила огляду коду:
  - Мінімум **одне** схвалення від члена команди для злиття **PR**.
  - Скасування старих схвалень при додаванні нових комітів.
  - Вирішення всіх коментарів перед злиттям обов'язкове.

- Робочий процес завдань:
  - Використання **GitHub Issues** для трекінгу завдань.
  - Налаштовано систему міток (**labels**) для категоризації.
  - Зв'язування **PR** з **Issues** через ключові слова (Fixes #123).
  - **CI pipeline** налаштований перевіряти всі зміни автоматично.

## 5. Coding Standards & Quality Policy

- ### Code Style Guides - Flutter

  - Дотримання [Dart Style Guide](https://dart.dev/guides/language/effective-dart/style).

  - Використано **265+ правил лінтингу** з `package:flutter_lints/flutter.yaml`, включно <br> з OWASP security-перевірками:  
    `avoid_dynamic_calls`, `unsafe_html`, `cancel_subscriptions`, `close_sinks`.

  - Активовано строгий типізаційний контроль (`strict-casts`, `strict-inference`, `strict-raw-types`).

  - Форматування здійснюється за допомогою `dart format`, а статичний аналіз — через `flutter analyze`.

  - Винятки дозволені лише для згенерованих файлів (`*.g.dart`, `*.freezed.dart`, `*.mocks.dart`).  

- ### Code Style Guides - Node.js

  - Використання **ESLint v9 (flat config)** з **280+ правилами**, основаними <br> на [Google Style Guide](https://google.github.io/styleguide/jsguide.html).

  - Основні правила:  
    - Безпека (OWASP): `no-eval`, `no-implied-eval`, `no-new-func`, `strict`.  
    - Якість коду: `consistent-return`, `no-shadow`, `require-await`.

  - Форматування — через `Prettier`, лінтинг — через `eslint`.

- Заборонено зливати зміни без успішного проходження CI-перевірок.

- ### Обов’язкові інструменти

  | Тип інструменту | Flutter | Node.js |
  |-----------------|----------|---------|
  | **Linter** | `flutter analyze` (265+ правил) | `eslint` (280+ правил) |
  | **Formatter** | `dart format` | `prettier` |
  | **CI перевірки** | GitHub Actions (`.github/workflows/code-quality.yml`) | GitHub Actions |
  | **Тестування** | `flutter test` | `jest` або `mocha` |
  | **Security Rules** | OWASP (Flutter analyzer) | OWASP (ESLint) |

<br>

- ### Code Review процес

  - Усі зміни надходять через **Pull Request (PR)**.

  - Кожен PR проходить **мінімум два незалежні рев’ю**.

  -  Перед **review** автоматично виконується:
    - `flutter analyze` + `eslint`;
    - запуск тестів у CI;
    - перевірка форматування (`dart format`, `prettier`).  
  - **Reviewers** перевіряють:
    - відповідність стилю коду;
    - наявність тестів і покриття ≥ 70%;
    - дотримання правил безпеки (OWASP);
    - оновлення документації при змінах API чи логіки.  
  - **PR Checklist:**
    - Код чистий і відформатований;  
    - Без критичних lint-помилок;  
    - Тести успішно пройдені;  
    - Security checks — пройдені;  
    - Документацію оновлено.  

- ### Definition of Done (DoD)

  - Код успішно проходить усі лінтери без помилок.
  - Покриття тестами не нижче **70%**.
  - Всі перевірки CI/CD завершені успішно.
  - PR отримав **щонайменше два approve**.
  - Усі OWASP-порушення усунені.
  - Документація актуалізована.
  - Немає блокуючих issues у статичному аналізі.

- ### Робота з технічним боргом

  - Технічний борг реєструється у backlog із тегом `tech-debt`.  
  - Визначаються відповідальний і дедлайн усунення.  
  - Quality Lead забезпечує те, щоб борг не перевищував допустимий рівень.  

- ### Узгодження та підтримка

  - Зміни до політики якості узгоджуються через PR-review.  
  - Quality Lead здійснює **щотижневий аудит** статичного аналізу.  
  - Результати CI/CD зберігаються як артефакти для контролю.  
  - Політика оновлюється відповідно до стандартів **OWASP**, **Dart** і **Google Style Guide**.
