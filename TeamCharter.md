## 4.1. Співпраця та робочий процес - Організація середовища GitHub

- Стратегія гілкування:
  - GitHub Flow:
    - Основна гілка `main` - завжди стабільна і готова до розгортання.
    - Присутні feature гілки: (`feature/feature-name`) для нових функцій.
    - Присутні bug fix гілки (`fix/bug-name`) для виправлень.
  - Захист гілки `main` налаштовано через GitHub Rulesets.
  - Зміни гілки `main` проводяться тільки через **Pull Requests** з обов'язковим <br> **code review**.
- Практики комітів:
  - Для найменування та опису комітів використовується лише англійська мова.
  - Формат `<type>: <description>` (наприклад: `feat: add user login`, <br> `fix: resolve validation issue`, `docs: update API documentation`, <br> `refactor: simplify database queries`).
  - Для будь-яких дій (змін) повинні бути додані лаконічні описові повідомлення, <br> що пояснюють їх суть.
- Команди **Git** для розробки:

```bash
  # Створення нової feature гілки
  git checkout main
  git pull origin main
  git checkout -b feature/your-feature-name

  # Робота з змінами
  git add .
  git commit -m "feat: add your feature description"
  git push origin feature/your-feature-name

  # Після злиття PR - очищення
  git checkout main
  git pull origin main
  git branch -d feature/your-feature-name
```

- Правила огляду коду:

  - Мінімум **одне** схвалення від члена команди для злиття **PR**.
  - Скасування старих схвалень при додаванні нових комітів.
  - Вирішення всіх коментарів перед злиттям обов'язкове.

- Робочий процес завдань:
  - Використання **GitHub Issues** для трекінгу завдань.
  - Налаштовано систему міток (**labels**) для категоризації.
  - Зв'язування **PR** з **Issues** через ключові слова (Fixes #123).
  - **CI pipeline** налаштований перевіряти всі зміни автоматично.

## 4.2. Співпраця та робочий процес - Процеси співпраці та контролю

- Практики співпраці та контролю:
  - Рецензування користувацьких історій відбувається у форматі **Pull Request**:
    - Перед злиттям у `main`історії обов’язково переглядаються не менше, ніж <br> одним іншим членом команди.
  - Для перевірки нефункціональних вимог проводиться спільна сесія огляду, <br> під час котрої формулюються вимірювані критерії.
- Управління артефактами вимог:
  - Всі документи з вимогами зберігаються у каталозі `/docs/requirements/` <br> репозиторію.
  - `README.md` у корені проєкту завжди містить актуальні посилання. <br> Актуальність, котрих періодично (щотижнево) перевіряється керівником <br> документації та супроводжувачем репозиторію.
- Контроль та індивідуальна відповідальність:
  - Кожен член команди створює відео-презентацію в Loom (3–5 хв), де демонструє <br> свою роботу в ході певної лабораторної роботи:
    - Окреслює ролі, котрі активно виконував протягом ходу поточної роботи, та <br> завдання, за котрі несе відповідальність.
  - Окреслені завдання повинні співпадати з призначеними `issues` на `GitHub`.
  - Звіти-відео зберігаються в спеціальні папки типу `Labs/Lab01`.
  - За потреби відео переглядаються на спільній огляд-сесії. Оскарження відбувається, <br> згідно **пункту п'ятого** цього статуту.
  - Апелювання до викладачів відбувається лише після обговорення та за наявних доказів.
